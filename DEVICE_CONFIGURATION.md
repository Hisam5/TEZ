# Device Configuration Guide

This project is now **fully device-independent**! All hardcoded paths and hostnames have been eliminated.

## ✅ Fixed Issues

The following device-specific hardcoded values have been made configurable:

### 1. ROS Bridge & URDF Server URLs (panda_hmi.py)
- **Before:** Hardcoded to `ws://localhost:9090` and `http://localhost:8080/`
- **After:** Configurable via URL parameters (with sensible defaults for localhost)

### 2. Robot Frame Name (panda_hmi.py)
- **Before:** Hardcoded to `panda_link0` (wrong robot!)
- **After:** Defaults to `base_link` (configurable)

### 3. Absolute Paths in URDF Files ⚠️ **CRITICAL FIX**
- **Before:** `/home/hesham/ros2_ws/install/tm5_900/share/tm5_900/config/ros2_controllers.yaml`
- **After:** `$(find tm5_900)/config/ros2_controllers.yaml`

**Files Fixed:**
- `src/tm5_900/urdf/tm5_900.urdf`
- `src/tm5_900/urdf/tm5_withgripper.urdf`
- `src/tm5_900/textures/urdf/tm5_900.urdf`
- `src/tm5_900/textures/urdf/tm5_withgripper.urdf`

This ROS-standard path resolution ensures files are found regardless of:
- Installation location
- Username
- System path structure

## 🚀 How to Use On Another Machine

### Step 1: Clone Your Project
```bash
git clone <your-repo-url> ~/my_ros_ws
cd ~/my_ros_ws
```

### Step 2: Source Your ROS 2 Environment
```bash
source /opt/ros/humble/setup.bash  # or your ROS 2 distro
source ./install/setup.bash
```

### Step 3: Run It (Works Out-of-the-Box!)
```bash
# Local machine - uses defaults
ros2 launch tm5_900 bringup.launch.py

# Or with web GUI
python3 src/tm5_900/panda_hmi.py
```

## 🌍 Remote Setup (Different Host)

If accessing from another machine, configure via URL parameters:

```
http://your-machine-ip:5000/?ros_url=ws://192.168.1.100:9090&urdf_url=http://192.168.1.100:8080&frame=base_link
```

**Query Parameters:**
- `ros_url`: ROS 2 bridge websocket URL (default: `ws://localhost:9090`)
- `urdf_url`: URDF server HTTP URL (default: `http://localhost:8080/`)
- `frame`: Robot TF frame name (default: `base_link`)

## 📝 Understanding the Fixes

### Why $(find package_name)?

ROS uses `$(find)` and `${find:}` substitutions to resolve package paths at runtime. This is evaluated by:
1. Loading the ROS environment
2. Finding the package in your ROS_PACKAGE_PATH
3. Substituting the absolute path for that installation

**Advantages:**
- ✅ Works on any user's machine
- ✅ Works on any system (Linux, macOS, Windows)
- ✅ Works with different ROS distributions
- ✅ Works in containers/VMs with mounted workspaces

### Why URL Parameters?

The web GUI couldn't access environment variables directly from the browser. URL parameters allow:
- Device-specific configuration without code changes
- Remote access support
- Easy testing of different configurations

## 🔍 Troubleshooting

### "Cannot find package tm5_900"
- **Cause:** ROS environment not sourced
- **Fix:** 
  ```bash
  source ~/ros2_ws/install/setup.bash
  ```

### "Cannot connect ROS bridge"
- **Cause:** ROS bridge not running on remote host
- **Fix:**
  ```bash
  # On the robot/server host
  ros2 launch rosbridge_server rosbridge_websocket_launch.xml
  ```

### "Cannot load URDF from server"
- **Cause:** URDF server not running
- **Fix:**
  ```bash
  # Start a simple HTTP server
  cd ~/ros2_ws/src/tm5_900
  python3 -m http.server 8080
  ```

### Wrong frame displayed in RViz
- **Cause:** Wrong TF frame name
- **Fix:** 
  ```bash
  # List available frames
  ros2 run tf2_tools view_frames.py
  
  # Use correct frame in URL
  ?frame=your_actual_frame_name
  ```

## 🛠️ For Project Maintainers

### Important Notes

1. **URDF Generation**: These URDF files are autogenerated from xacro sources. If you edit the xacro files, regenerate URDFs:
   ```bash
   cd src/tm5_900/urdf
   xacro tm5_900.urdf.xacro > tm5_900.urdf
   xacro tm5_900.urdf.xacro -o tm5_withgripper.urdf
   ```

2. **Always Use ROS Path Resolution**: When adding new paths:
   - ✅ Use `$(find package_name)/path` in URDF/config files
   - ✅ Use `get_package_share_directory()` in Python launch files
   - ❌ Never hardcode `/home/username/...` paths

3. **Testing Portability**: Test your project with:
   ```bash
   # Simulate different installation path
   colcon build --symlink-install
   rm -rf install
   colcon build
   ```
